# LightMem: Lightweight and Efficient Memory-Augmented Generation

> Jizhan Fang, Xinle Deng, Haoming Xu, Ziyan Jiang, Yuqi Tang, Ziwen Xu, Shumin Deng, Yunzhi Yao, Mengru Wang, Shuofei Qiao, Huajun Chen, Ningyu Zhang
> Zhejiang University / NUS / Nanjing University | ICLR 2026 | [https://github.com/zjunlp/LightMem](https://github.com/zjunlp/LightMem)

## 📄 核心贡献

- 提出 **LightMem**：受人类记忆Atkinson-Shiffrin模型启发的轻量高效记忆系统，发表于ICLR 2026
- 核心创新：**三阶段记忆处理**——感觉记忆(预压缩) → 主题感知短期记忆 → 睡眠时间更新的长期记忆
- 在LongMemEval和LoCoMo上同时实现SOTA性能和极致效率：**token使用减少106倍，API调用减少310倍**

## 🔬 方法论（大白话讲解）

### 系统架构：人类记忆三阶段

LightMem 直接模仿人类记忆的工作方式：

| 阶段 | 人类记忆 | LightMem 实现 |
|------|----------|---------------|
| **感觉记忆** | 眼睛看到大量信息，大脑快速过滤无关的 | 轻量预压缩(LLMLingua-2) + 主题分割 |
| **短期记忆** | 工作记忆，按主题组织信息 | 主题感知STM：按话题分组，触发摘要 |
| **长期记忆** | 睡眠时巩固记忆，去重、整理 | 离线并行更新：测试时只插入，离线时整理 |

### Light1：感觉记忆（预压缩）

**问题**：原始对话包含大量冗余信息（问候、寒暄、重复），直接喂给LLM浪费token。

**解决方案**：
1. **Token预压缩**：用LLMLingua-2对每个token计算"保留概率"，只保留高概率token
   - 基于交叉熵：信息量大的token（难以预测的）更可能被保留
   - 压缩率r可调（通常0.5-0.8），保留50%-80%的token
2. **主题分割**：混合注意力+相似度方法
   - 注意力边界：用压缩模型的注意力矩阵找话题切换点
   - 相似度边界：用语义相似度验证（避免注意力噪声）
   - 最终边界 = 两种方法的交集

**大白话**：先用一个轻量模型"扫一遍"对话，把不重要的词删掉，然后按话题把内容分成一块一块的。

### Light2：主题感知短期记忆

**问题**：固定按turn或session分割，要么太细（频繁调用LLM），要么太粗（话题混杂导致记忆不准确）。

**解决方案**：
- 建立 `{topic, message_turns}` 索引结构
- 只有当STM缓冲区token数达到阈值(th)时，才调用LLM生成摘要
- 摘要存入LTM，原始内容保留在STM中用于短期上下文

**优势**：topic粒度最小化API调用，同时保持摘要准确性。

### Light3：长期记忆 + 睡眠时间更新

**问题**：在线实时更新记忆会引入推理延迟，且顺序更新效率低。

**解决方案**：

**测试时（软更新）**：
- 新记忆条目直接插入LTM，不触发在线更新
- 为每个条目计算更新队列Q(ei)：找最相似的、时间戳更新的记忆条目
- **只插入不覆盖**，保留完整上下文

**离线时（硬更新/睡眠）**：
- 并行处理所有记忆条目的更新
- 去重、合并矛盾、抽象总结
- 解耦了昂贵的记忆维护和实时推理

**案例对比**：
```
历史1: {周一2PM: 用户计划去东京}
历史2: {周一4PM: 用户询问去京都的火车}

硬更新（传统）: 覆盖为"用户计划京都之旅" → 东京信息丢失
LightMem软更新: 追加为"东京之旅 + 京都询问" → 完整上下文保留
```

### 效率分析

| 方法 | 运行时复杂度 | LightMem |
|------|-------------|----------|
| 传统记忆系统 | O(N) | O(N·rxT/th) |
| API调用次数 | N次摘要 + N次更新 | N·rxT/th次摘要 + 离线更新 |

其中rxT是压缩后token数，th是STM阈值。当th=512, r=0.7时，API调用减少约7倍。

## 📊 实验结果

### 实验设置
- 数据集：LongMemEval-S、LoCoMo
- LLM骨干：GPT-4o-mini、Qwen3-30B-A3B-Instruct、GLM-4.6
- 评估：增量对话输入（逐轮喂入，模拟真实场景）

### LongMemEval 结果

| 方法 | ACC (GPT) | ACC (Qwen) | Token (GPT) | API Calls (GPT) |
|------|-----------|------------|-------------|-----------------|
| FullText | 56.80% | 54.80% | 105K | - |
| NaiveRAG | 61.00% | 60.80% | - | - |
| A-MEM | 62.60% | 65.20% | 1,606K | 987 |
| MemoryOS | 44.80% | 49.60% | 2,992K | 2,938 |
| Mem0 | 53.61% | 39.51% | 1,153K | 812 |
| **LightMem** | **68.64%** | **70.20%** | **28K** | **18** |

- ACC：LightMem 68.64% vs A-MEM 62.60%（+9.6%）
- Token：28K vs A-MEM 1,606K（**减少57倍**）
- API调用：18次 vs A-MEM 987次（**减少55倍**）

### LoCoMo 结果

| 方法 | ACC (GPT) | ACC (Qwen) | Token (GPT) | API Calls (GPT) |
|------|-----------|------------|-------------|-----------------|
| FullText | 71.83% | 74.87% | - | - |
| A-MEM | 64.16% | 56.10% | 1,149K | 1,175 |
| MemoryOS | 58.25% | 61.04% | 287K | 553 |
| Mem0 | 61.69% | 43.31% | 1,693K | 1,602 |
| **LightMem** | **72.99%** | **72.60%** | **85K** | **30** |

- ACC：LightMem 72.99% vs FullText 71.83%（略高）
- Token：85K vs A-MEM 1,149K（**减少13.5倍**）
- API调用：30次 vs A-MEM 1,175次（**减少39倍**）

### 纯在线测试时成本

仅考虑推理阶段（不含离线更新）：
- GPT：token减少**106倍**，API调用减少**159倍**
- Qwen：token减少**117倍**，API调用减少**310倍**

### 消融实验

| 配置 | GPT ACC | Qwen ACC |
|------|---------|----------|
| 有主题分割 | 68.64% | 73.20% |
| 无主题分割 | 62.34% | 67.80% |
| **提升** | **+6.3%** | **+5.4%** |

主题分割模块对性能有显著贡献。

## ⚠️ 局限性

- **依赖预压缩模型质量**：LLMLingua-2的压缩质量直接影响后续记忆构建
- **离线更新需要额外基础设施**：需要部署独立的离线更新服务
- **压缩率选择敏感**：不同任务的最优压缩率不同，需要调参
- **仅在对话任务验证**：未测试代码、文档等非对话场景
- **STM阈值选择影响性能**：th太大或太小都会影响效果
