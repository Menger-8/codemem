# A-Mem: Agentic Memory for LLM Agents

> Wujiang Xu, Zujie Liang, Kai Mei, Hang Gao, Juntao Tan, Yongfeng Zhang
> Rutgers University / AIOS Foundation | 2025 | [https://github.com/WujiangXu/AgenticMemory](https://github.com/WujiangXu/AgenticMemory)

## 📄 核心贡献

- 提出 **A-Mem**：借鉴 Zettelkasten（卡片盒笔记法）思想的智能体记忆系统，让LLM**自主组织**记忆，而非依赖预定义的存储结构
- 核心创新：**链接生成 + 记忆演化**——新记忆自动与历史记忆建立关联，同时触发已有记忆的更新优化
- 在6种基础模型上验证，token效率提升85-93%，同时性能超越 MemGPT 等SOTA

## 🔬 方法论（大白话讲解）

### 系统架构

A-Mem 的灵感来自 **Zettelkasten 笔记法**——一种知识管理方法，核心思想是"每条知识写一张原子卡片，卡片之间通过链接形成知识网络"。

整体流程分4步：

```
新交互 → 笔记构造 → 链接生成 → 记忆演化 → [检索时] 相关记忆检索
```

### 笔记构造（Note Construction）

每条记忆被构造成一张"笔记"，包含6个属性：

| 属性 | 作用 | 由谁生成 |
|------|------|----------|
| `ci` (Content) | 原始交互内容 | 用户/环境 |
| `ti` (Timestamp) | 时间戳 | 系统 |
| `Ki` (Keywords) | 关键概念 | LLM |
| `Gi` (Tags) | 分类标签 | LLM |
| `Xi` (Context) | 丰富的上下文描述 | LLM |
| `ei` (Embedding) | 密集向量表示 | 文本编码器 |

**大白话**：不只是存原始对话，而是让LLM"消化"后生成关键词、标签和上下文描述，再用编码器转成向量。这样每条记忆既有语义理解（LLM生成的文本），也有计算效率（向量检索）。

### 链接生成（Link Generation）

当新笔记加入系统时：

1. 用新笔记的向量去检索历史笔记中最相似的 top-k 条
2. 把新笔记和这些候选历史笔记一起给LLM分析
3. LLM 判断是否存在有意义的关联，如果有就建立链接
4. 每张笔记维护一个链接列表 `Li = {mi, ..., mk}`

**大白话**：类似你写笔记时发现"这个概念和之前那个笔记有关联"，于是在两张卡片之间画一条线。A-Mem 让LLM自动完成这个"发现关联"的过程。

### 记忆演化（Memory Evolution）

这是 A-Mem 最独特的部分——当新记忆加入时，**已有记忆也会被更新**：

1. 新笔记的邻居集（top-k相似记忆）被找出
2. 对于每个邻居笔记 `mj`，LLM 分析新笔记 `mn` 是否提供了新的上下文信息
3. 如果有，LLM 会更新 `mj` 的上下文描述、关键词或标签

**大白话**：人类学习新知识后，会重新理解旧知识（"原来那个概念可以这样理解"）。A-Mem 模拟了这个过程——新记忆的加入会"反哺"旧记忆，让整个记忆网络持续进化。

### 检索机制

标准的向量检索：查询文本编码 → 与所有笔记向量计算余弦相似度 → 取 top-k。检索到的相关笔记会自动带上其链接的笔记（类似 Zettelkasten 中"找到一张卡片，顺着链接找到更多相关卡片"）。

### 关键设计选择

- **为什么用Zettelkasten而不是知识图谱？** 图数据库（如Neo4j）需要预定义schema，而Zettelkasten更灵活——笔记之间的关系是动态生成的，不需要预设节点类型和边类型
- **为什么需要记忆演化？** 静态记忆系统的问题是"旧记忆的上下文描述可能过时"，演化机制让记忆网络能自我完善
- **为什么token效率高？** 只检索top-k相关笔记（~1200 token），而不是全量对话（~16900 token）

## 📊 实验结果

### 实验设置
- 数据集：LoCoMo（长对话QA）、DialSim（实时对话模拟）
- 评估指标：F1、BLEU-1、ROUGE-L/METEOR、SBERT Similarity
- 基础模型：GPT-4o-mini, GPT-4o, Qwen2.5 (1.5b/3b), Llama 3.2 (1b/3b)

### LoCoMo 关键结果（GPT-4o-mini）

| 方法 | Multi-Hop F1 | Temporal F1 | Open Domain F1 | Single Hop F1 | Avg Token |
|------|-------------|-------------|----------------|---------------|-----------|
| LoCoMo | 25.02 | 18.41 | 12.04 | 40.36 | 16,910 |
| MemGPT | 26.65 | 25.52 | 9.15 | 41.04 | 16,977 |
| **A-MEM** | **27.02** | **45.85** | **12.14** | **44.65** | **2,520** |

- A-Mem 时间推理能力最强（45.85 vs MemGPT 25.52，提升80%）
- Token效率：仅需2,520 token（vs MemGPT 16,977），**减少85%**

### 跨模型一致性

A-Mem 在所有6种基础模型上均优于 baseline，尤其在小模型上优势更大：
- Qwen2.5-1.5b：A-Mem 多跳F1=18.23 vs MemGPT 10.44（+75%）
- Llama 3.2-3b：A-Mem 多跳F1=17.44 vs MemGPT 5.32（+228%）

### 消融实验

| 方法 | Multi-Hop F1 | Temporal F1 |
|------|-------------|-------------|
| w/o LG & ME | 9.65 | 24.55 |
| w/o ME (只有链接) | 21.35 | 31.24 |
| A-MEM (完整) | **27.02** | **45.85** |

- 链接生成贡献最大（从9.65提升到21.35）
- 记忆演化进一步提升（21.35→27.02）
- 两者结合效果最佳

### 可扩展性

| 记忆规模 | A-Mem 检索时间 | MemoryBank | ReadAgent |
|----------|---------------|------------|-----------|
| 1,000 | 0.31µs | 0.24µs | 43.62µs |
| 100,000 | 1.40µs | 0.78µs | 6,682µs |
| 1,000,000 | 3.70µs | 1.91µs | 120,070µs |

A-Mem 检索时间几乎不随规模增长（O(1)级检索），远优于 ReadAgent。

## ⚠️ 局限性

- **依赖LLM质量**：笔记构造、链接生成、记忆演化全部依赖LLM，不同LLM生成的质量差异大
- **记忆演化可能引入噪声**：新记忆触发旧记忆更新，如果LLM判断错误，可能"污染"已有记忆
- **仅在对话任务上验证**：未测试代码生成、文档分析等场景
- **未与 Mem0 直接对比**：Mem0 的 J-score 指标未纳入对比
- **多次LLM调用的延迟**：每次记忆操作需要3次LLM调用（构造+链接+演化），虽然token少但调用次数多
