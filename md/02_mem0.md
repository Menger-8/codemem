# Mem0: Building Production-Ready AI Agents with Scalable Long-Term Memory

> Prateek Chhikara, Dev Khant, Saket Aryan, Taranjeet Singh, Deshraj Yadav
> mem0.ai | 2025 | [https://mem0.ai/research](https://mem0.ai/research)

## 📄 核心贡献

- 提出 **Mem0** 架构：通过"提取→更新"两阶段流水线，从对话中动态提取关键事实并维护一致性记忆库
- 提出 **Mem0g** 增强版：引入图结构记忆（知识图谱），捕获实体间的复杂关系，提升时间推理能力
- 在 LOCOMO 基准上系统对比6类baseline，Mem0 在准确率上超越 OpenAI 26%，同时 **p95延迟降低91%、token成本节省90%以上**

## 🔬 方法论（大白话讲解）

### 系统架构

Mem0 的核心思路：**不存储原始对话，只提取关键事实**。这样回答问题时只需要看少量事实，而不是整段对话。

两阶段流水线：

**第一阶段：记忆提取（Extraction）**
- 输入：新对话消息 + 最近10条消息上下文 + 对话摘要（异步更新）
- LLM 从对话中提取一组"关键事实"（如"用户是素食主义者"、"用户住在旧金山"）
- 关键：不是逐句提取，而是结合全局摘要理解"什么信息值得记住"

**第二阶段：记忆更新（Update）**
- 对每个新提取的事实，用向量检索找到最相似的已有记忆
- LLM 决定执行以下4种操作之一：
  - **ADD**：没有重复记忆，新增一条
  - **UPDATE**：已有类似记忆，补充新信息
  - **DELETE**：新信息与旧记忆矛盾，删除旧的
  - **NOOP**：不需要操作（冗余信息）
- 使用函数调用（Tool Call）机制让 LLM 自己选择操作

### Mem0g：图结构增强

在 Mem0 基础上增加知识图谱层：

- **实体** = 节点（如 Alice、旧金山），带类型和嵌入向量
- **关系** = 有向边（如 lives_in、prefers），三元组形式存储
- 使用 Neo4j 图数据库存储
- **双路检索**：实体中心法（从关键实体出发探索关系子图）+ 语义三元组法（查询向量与所有三元组匹配）

### 关键设计选择

- **为什么用事实提取而不是存原始文本？** 原始对话噪声大、冗余多。提取后的事实更精炼，token消耗仅为原始对话的1/4（7k vs 26k）
- **为什么不用分类器判断记忆操作？** 直接让 LLM 用函数调用选择操作，利用 LLM 的语义理解能力，避免训练额外分类器
- **异步摘要更新**：摘要生成独立于主流程，不引入延迟

## 📊 实验结果

### 实验设置
- 数据集：LOCOMO（长对话基准）
- 问题类型：单跳、多跳、开放域、时间推理
- 评估指标：F1、BLEU-1、LLM-as-a-Judge (J)
- Baseline：6类（已发表系统、RAG、全上下文、开源方案、OpenAI、商业平台Zep）

### 关键结果

| 方法 | Overall J | 单跳J | 多跳J | 开放域J | 时间J |
|------|-----------|-------|-------|---------|-------|
| RAG (best) | 60.97 | - | - | - | - |
| MemGPT | - | - | 9.15 | 41.04 | 25.52 |
| A-Mem | 48.38 | 39.79 | 18.85 | 54.05 | 49.91 |
| OpenAI | 52.90 | 63.79 | 42.92 | 62.29 | 21.71 |
| Zep | 65.99 | 61.70 | 41.35 | **76.60** | 49.31 |
| **Mem0** | **66.88** | **67.13** | **51.15** | 72.93 | 55.51 |
| **Mem0g** | **68.44** | 65.71 | 47.19 | 75.71 | **58.13** |
| Full-context | 72.90 | - | - | - | - |

- Mem0 比 OpenAI 高 **26%**（J=66.88 vs 52.90）
- Mem0g 在时间推理上优势最大（58.13 vs OpenAI 21.71）
- 多跳推理 Mem0 最强（51.15），图结构反而没有帮助

### 部署效率

| 方法 | p95延迟 | 搜索延迟p50 | 记忆token |
|------|---------|-------------|-----------|
| Full-context | 17.12s | - | 26,031 |
| Mem0 | **1.44s** | 0.15s | 7,000 |
| Mem0g | 2.59s | 0.48s | 14,000 |
| Zep | 2.93s | 0.51s | 600,000+ |

- Mem0 延迟仅全上下文的 **8%**
- Zep 的记忆token是 Mem0 的 **85倍**

## ⚠️ 局限性

- **全上下文仍略优**：Full-context J=72.90，Mem0g=68.44，准确性仍有差距
- **图结构对多跳无帮助**：Mem0g 多跳J=47.19，反而低于 Mem0 的51.15，图的导航开销抵消了结构化优势
- **依赖LLM提取质量**：事实提取的准确性完全取决于LLM能力，可能出现幻觉或遗漏
- **仅在英文对话上验证**：未覆盖多语言和非对话场景（如代码、文档）
- **时间推理依赖显式时间戳**：OpenAI 因缺少时间戳表现极差（J=21.71），Mem0 系统需要显式提示提取时间信息
